## 2. 完整业务逻辑链条

### 2.1 逻辑流程图

```
客户累计消费(totalSpent)
        ↓
[客户等级计算] getCustomerLevel(totalSpent)
        ↓
客户等级(level) → [折扣率计算] getDiscountRate(totalSpent)
        ↓
折扣率(discountRate)
        ↓
┌─────────────────────────────────────────┐
│  创建销售单时(salesOrderStore.add)      │
│  1. 保存 customerLevel 快照             │
│  2. 保存 discountRate 快照              │
│  3. 计算 originalAmount (原价总额)      │
│  4. 计算 totalAmount = 原价 × 折扣率    │
│  5. 每个商品明细应用折扣                │
└─────────────────────────────────────────┘
        ↓
销售单状态: pending（待审核）
        ↓
[审核] updateStatus('approved')
        ↓
┌─────────────────────────────────────────┐
│  1. 创建应收账款 financeStore.addReceivable   │
│     金额 = 销售单.totalAmount（已折后）  │
└─────────────────────────────────────────┘
        ↓
销售单状态: approved（已审核）
        ↓
[确认出库] updateStatus('completed')
        ↓
┌─────────────────────────────────────────┐
│  1. 扣减库存 productStore.updateStock   │
│  2. 记录库存出入库流水                  │
│  3. 更新客户累计消费                    │
│     customer.totalSpent += totalAmount  │
│     重新计算 customer.level             │
└─────────────────────────────────────────┘
        ↓
销售单状态: completed（已完成）
        ↓
┌─────────────────────────────────────────┐
│  利润统计(reports页面)                   │
│  销售额 = Σ 已完成订单.totalAmount      │
│  成本 = Σ (商品成本 × 数量)             │
│  毛利润 = 销售额 - 成本                 │
└─────────────────────────────────────────┘
```

### 2.2 关键节点代码分析

#### 节点1：客户等级动态计算

**代码位置**：[helpers.ts](file:///Users/lilixian/工作相关/AI/ai-apps-workspace/02.work session/solo-0605/source code /pro-28/pro-28/src/lib/utils/helpers.ts#L15-L32)

```typescript
export function getCustomerLevel(totalSpent: number): CustomerLevel {
  for (let i = CUSTOMER_LEVEL_CONFIGS.length - 1; i >= 0; i--) {
    const config = CUSTOMER_LEVEL_CONFIGS[i];
    if (totalSpent >= config.minSpent) {
      return config.level;
    }
  }
  return 'bronze';
}

export function getDiscountRate(totalSpent: number): number {
  const level = getCustomerLevel(totalSpent);
  return getCustomerLevelConfig(level).discountRate;
}
```

**分析**：
- 客户等级完全由 `totalSpent` 决定，是一个纯函数
- 每次调用都会根据当前累计消费重新计算
- 折扣率通过等级间接获取

#### 节点2：销售单创建时的折扣应用

**代码位置**：[salesOrders.svelte.ts](file:///Users/lilixian/工作相关/AI/ai-apps-workspace/02.work session/solo-0605/source code /pro-28/pro-28/src/lib/stores/salesOrders.svelte.ts#L16-L47)

```typescript
add(order: Omit<SalesOrder, ...>) {
  // 创建时获取当前折扣率和客户等级
  const discountRate = customerStore.getDiscountRate(order.customerId);
  const customerLevel = customerStore.getLevel(order.customerId);

  // 每个商品应用折扣
  const discountedItems = order.items.map((item, idx) => ({
    ...item,
    id: idx + 1,
    price: Math.round(item.price * discountRate * 100) / 100,
    amount: Math.round(item.amount * discountRate * 100) / 100
  }));

  const originalAmount = order.items.reduce((sum, i) => sum + i.amount, 0);
  const totalAmount = Math.round(originalAmount * discountRate * 100) / 100;

  // 保存快照到销售单
  const newOrder: SalesOrder = {
    ...order,
    items: discountedItems,
    customerLevel,      // 快照
    discountRate,       // 快照
    originalAmount,     // 原价
    totalAmount,        // 折后价
    // ...
  };
  this.items.unshift(newOrder);
  return newOrder;
}
```

**关键设计**：
- ✅ **折扣率和客户等级在创建时快照保存**，不会随后续客户等级变化而变化
- ✅ **商品明细单独应用折扣**，每行的 `price` 和 `amount` 都是折后价
- ✅ **保存原价和折后价**，便于后续统计和对账

## 3. 时序依赖问题详细分析

### 3.1 问题一：待审核订单的折扣是否会随客户等级变化？

**答案：不会变化**

**原因分析**：

1. **折扣在创建时已快照保存**
   - 销售单创建时，`discountRate` 和 `customerLevel` 就已经确定并保存到销售单中
   - 销售单的 `items` 中的每个商品的 `price` 和 `amount` 也已经用当时的折扣率计算完成
   - `originalAmount`（原价）和 `totalAmount`（折后价）也已保存

2. **状态变更不重新计算折扣**
   - 审核（`approved`）时，直接使用已保存的 `order.totalAmount` 创建应收账款
   - 确认出库（`completed`）时，直接使用已保存的 `order.totalAmount` 更新累计消费
   - 整个状态流转过程中，没有任何代码会重新计算折扣率

3. **代码验证**
   - [salesOrders.svelte.ts](file:///Users/lilixian/工作相关/AI/ai-apps-workspace/02.work session/solo-0605/source code /pro-28/pro-28/src/lib/stores/salesOrders.svelte.ts#L49-L64) 中的 `updateStatus` 方法只修改 `status` 和 `updatedAt`，不涉及折扣重算

**场景示例**：
```
客户A当前累计消费：45,000 → 青铜等级 → 95折
1月1日：创建销售单SO001，原价10,000 → 折后9,500，状态pending
1月2日：客户A完成另一笔订单，累计消费变为55,000 → 升级为白银 → 90折
1月3日：审核SO001 → 应收账款金额仍是9,500（创建时的折后价）
1月4日：SO001出库 → 客户累计消费 += 9,500 → 变为64,500
```

### 3.2 问题二：已完成订单的利润统计是否会因客户等级变化而变动？

**答案：不会变动**

**原因分析**：

1. **利润计算基于历史快照**
   - 销售额使用销售单中已保存的 `totalAmount`（折后价快照）
   - 该值在销售单创建时就已确定，不会随后续客户等级变化而变化
   - 成本计算基于商品成本和销售数量，与客户等级完全无关

2. **历史数据的不可变性**
   - 已完成的销售单没有任何代码会修改其 `totalAmount` 或 `items`
   - 客户等级变化只会影响**后续新建**的销售单，不会追溯修改历史订单
   - 报表统计是实时计算，但计算的数据源是不变的历史快照

3. **代码验证**
   - [reports/+page.svelte](file:///Users/lilixian/工作相关/AI/ai-apps-workspace/02.work session/solo-0605/source code /pro-28/pro-28/src/routes/reports/+page.svelte#L62-L71) 中的利润计算完全基于销售单中已保存的数据

**场景示例**：
```
1月：客户A是青铜（95折），完成订单SO001，原价10,000，折后9,500
     商品成本6,000 → 利润3,500
     1月利润统计：销售额9,500，成本6,000，利润3,500

2月：客户A升级为黄金（85折），完成订单SO002，原价10,000，折后8,500
     商品成本6,000 → 利润2,500
     重新统计1月利润：销售额仍为9,500，利润仍为3,500（不变）
     2月利润统计：销售额8,500，成本6,000，利润2,500
```

## 4. 代码设计优点总结

### 4.1 快照设计模式

系统采用了**快照（Snapshot）设计模式**，这是处理此类时序依赖问题的最佳实践：

| 设计决策 | 优点 |
|---------|------|
| 销售单保存 `customerLevel` 快照 | 确保订单显示的客户等级与创建时一致 |
| 销售单保存 `discountRate` 快照 | 明确记录该订单享受的折扣政策 |
| 销售单保存 `originalAmount` | 便于对账和审计，知道原价多少 |
| 销售单保存 `totalAmount` | 实际结算金额，所有下游计算都基于此 |
| 每个商品明细单独应用折扣 | 行级数据一致，便于单品利润分析 |

### 4.2 单向因果链设计

```
客户累计消费 → 客户等级 → 折扣率 → 新销售单 → 订单完成 → 客户累计消费增加
```

- 因果关系清晰，没有循环依赖
- 历史数据不会被追溯修改
- 每个环节的输入输出明确

## 6. 总结

### 6.1 核心结论

| 问题 | 结论 | 依据 |
|------|------|------|
| 待审核订单折扣是否会随客户等级变化？ | ❌ 不会变化 | 折扣在创建时已快照保存，状态变更不重算 |
| 已完成订单利润统计是否会变动？ | ❌ 不会变动 | 利润基于历史快照数据计算，不追溯修改 |

### 6.2 设计亮点

1. ✅ **快照模式**：折扣相关数据在订单创建时全部快照保存
2. ✅ **单向因果**：数据流向清晰，无循环依赖
3. ✅ **数据一致**：应收账款、累计消费、利润统计都基于同一快照值
4. ✅ **边界明确**：客户等级变化只影响后续新订单，不影响历史数据

### 6.3 改进优先级

| 改进项 | 优先级 | 影响 |
|-------|--------|------|
| 统一折扣率配置来源 | P0 高 | 防止配置不一致导致的计算错误 |
